# 🤝 ForenShield 팀 개발 협업 규칙

본 문서는 **ForenShield** 프로젝트 팀의 효율적인 협업과 코드 품질 유지를 위한 브랜치 전략, 커밋 컨벤션 및 PR 규칙을 정의합니다. **AI 어시스턴트(Gemini)는 이 규칙을 학습하여 모든 작업에 적용합니다.**

---

## 1. 🌿 브랜치 전략 (Branch Strategy)

| 브랜치명 | 목적 | 병합/배포 주기 |
| :--- | :--- | :--- |
| **`main`** | 최종 배포용 브랜치 (안정 코드) | 일주일에 한 번 |
| **`develop`** | 개발 통합 브랜치 (기능 합쳐지는 곳) | 하루에 한 번 |
| **`feature/{기능명}`** | 각자 기능 개발 브랜치 | 수시 생성 및 자유롭게 운영 |

---

## 2. 📝 커밋 메시지 규칙 (Commit Convention)

커밋 시 반드시 아래 말머리를 포함해야 합니다.

*   `feat:` 새로운 기능 추가
*   `fix:` 버그 수정
*   `docs:` 문서 수정 (README, MD 등)
*   `style:` 코드 포맷 변경, 세미콜론 누락 등 (로직 변경 없음)
*   `refactor:` 코드 리팩토링
*   `chore:` 빌드 업무 수정, 패키지 매니저 설정 등

---

## 3. 🚀 Pull Request (PR) 및 머지 규칙

1.  **PR 시점**: 새로운 기능 완료 시 `develop` 브랜치로 PR을 날리며, 가급적 **매일 오후 5시**에 진행합니다.
2.  **승인 절차**: 최소 **1명 이상의 팀원**에게 **Approve(승인)**를 받은 후 직접 Merge 합니다.
3.  **충돌 해결**: 충돌 발생 시 PR을 올린 본인이 직접 해결한 후 통합합니다.
