# AI 분석 PDF·공개 검증 단계별 구현 계획

> 기준일: 2026-07-14
>
> 대상: 분석 결과 PDF, 공개 QR 검증 페이지, 발행 버전·스냅샷
>
> 관련 요구사항: `RQ-DTL-084`~`RQ-DTL-088`, `RQ-CMP-104`
> 원칙: 실제 저장된 결과만 표시하고, 결측값·보류·실패를 임의 데이터로 대체하지 않는다.

---

## 1. 결론

현재 DB를 처음부터 다시 설계할 필요는 없다.

- **1차 PDF 재구성:** DB 스키마 변경 없이 진행 가능
- **2차 불변 발행 스냅샷:** 신규 스냅샷 저장 구조가 필요하므로 중간 규모 DB 변경
- **3차 CoC·외부 앵커·전자서명:** 실제 업무 규칙 확정 후 별도 확장

현재 `reports` 테이블에는 다음 필드가 이미 존재한다.

- 보고서 번호 `report_no`
- 검증 토큰·검증 코드
- 공개 접근 상태·발급·만료 시각
- 발행 상태 `DRAFT / ISSUED / SUPERSEDED`
- 보고서 버전
- 발행자·발행 시각·대체 시각
- 저장 경로
- 최종 PDF SHA-256
- 파일 크기·생성 시각

따라서 1차 목표는 기존 분석·보고서 데이터를 바탕으로 **오류 없이 읽히는 5페이지 핵심본**을 만드는 것이다. 발행 후 데이터가 변해도 PDF와 공개 페이지가 같은 값을 유지하도록 만드는 스냅샷은 2차에서 진행한다.

---

## 2. 최종 목표 구조

### 2.1 PDF

- 기본: **5페이지 핵심본**
- 추가 결과가 실제로 존재할 때만 조건부 부록 생성
- 실제 물리 페이지 수를 계산하여 `현재 페이지 / 전체 페이지` 표시
- 미리보기와 최종 발행본의 QR·URL·상태를 완전히 분리

### 2.2 공개 검증 페이지

- 주 기능: 보고서 발행 등록상태 확인
- 주 기능: 업로드한 PDF와 발행 등록본의 바이트 단위 동일성 확인
- 부가 기능: 공개가 승인된 범주형 분석 결과 요약
- 사건명, 원파일명, 담당자 실명, 이미지, 의심 구간, 숫자 점수는 기본 비공개

### 2.3 발행 데이터

- PDF와 공개 페이지가 동일한 불변 발행 스냅샷을 참조
- 이미 발행한 PDF와 스냅샷은 덮어쓰지 않음
- 수정은 개정본, 대체 또는 철회 관계로 관리

## 3. DB 영향도

| 단계 | DB 변경 | 규모 | 설명 |
|---|---:|---:|---|
| 0. 데이터 계약 확인 | 없음 | 없음 | 기존 필드의 의미와 누락 여부 확인 |
| 1A. PDF 발행 오류 수정 | 없음 | 작음 | 페이지 수, 푸터, 워터마크, QR 상태 수정 |
| 1B. 5페이지 핵심본 | 없음 | 중간 | 기존 `AnalysisResult`, 모듈 결과, 메타데이터 사용 |
| 1C. 실제 이미지 삽입 | 원칙적으로 없음 | 중간 | 발행 시 URL의 이미지 바이트를 가져와 PDF에 포함 |
| 1D. 공개 검증 문구·레이아웃 | 없음 | 작음 | 기존 발행 레지스트리 범위에서 개선 |
| 2A. 불변 발행 스냅샷 | 필요 | 중간 | 발행 시점 결과 JSON·표시 정책 저장 |
| 2B. 개정·대체·철회 확장 | 일부 필요 | 중간 | 기존 버전·대체 필드를 확장 |
| 2C. 무결성 결과 연결 | 없음 | 중간 | 기존 스냅샷 JSON에 CoC·매니페스트·증거 앵커 결과 추가 |
| 3. 기관 양식·선택적 서명 | 필요 가능 | 별도 | 수신기관이 명시적으로 요구하는 경우에만 진행 |

### 3.1 2차 DB 변경의 성격

2차는 기존 분석 테이블을 변경하는 작업이 아니라, 발행 당시 값을 복사해 보존하는 구조를 추가하는 작업이다. 권장 방향은 `reports` 테이블에 많은 결과 컬럼을 계속 추가하는 대신, 보고서별 불변 스냅샷을 별도 저장하는 것이다.

구현 전 `docs/database/erd.md`에서 확정해야 할 후보 구조:

- 보고서와 1:1인 발행 스냅샷
- 스냅샷 스키마 버전
- PDF 템플릿 버전
- 분석 결과 스냅샷 JSON
- 공개 결과 요약 JSON
- PDF에 채택된 구간·프레임·히트맵 참조와 해시
- 표시 단위·반올림 규칙·기준값
- 생성 시각·시간대

위 구조는 2A에서 `report_publication_snapshots` 테이블로 확정했다. 기존 분석 테이블은 변경하지 않고 보고서와 1:1인 불변 스냅샷을 추가한다.

## 4. 단계 0 — 데이터 계약 확인

### 목적

PDF 레이아웃을 만들기 전에 각 숫자와 상태의 실제 의미를 확정한다.

### 확인 항목

- `riskScore`의 범위와 산출 주체
- `confidenceScore`가 독립적으로 교정된 신뢰도인지, 위험 점수의 단순 보완값인지
- 최종 종합 모델명·버전
- 최종 종합 판정 기준값
- 딥페이크 영역 결과에 포함되는 모듈
- 영상 내용 조작 흔적 영역에 포함되는 모듈
- 보류·실패 모듈의 종합 결과 반영 방식
- 모델별 점수 범위와 기준값
- 대표 프레임 URL의 만료 방식과 백엔드 접근 가능 여부
- 히트맵 이미지가 현재 실제 저장되는지
- 얼굴 검출 수·판별 가능 트랙 수가 저장되는지

### 현재 확인된 상태

| 데이터 | 현재 상태 | 1차 처리 |
|---|---|---|
| 종합 점수·위험 등급·요약 | 저장됨 | 표시 가능 |
| 모듈명·모델명·버전·점수·탐지값 | 저장됨 | 영역별 그룹화 가능 |
| 모듈 타임라인·기준값 | 일부 저장됨 | 실제 전달된 모듈만 표시 |
| 프레임·클립·프레임쌍 위험도 | 저장 가능 | 통합 타임라인·상위 구간에 사용 |
| 대표 프레임 URL | 저장 가능 | 발행 시 실제 이미지 삽입 검토 |
| 대표 프레임 히트맵 | 현재 백엔드 DTO에 명시 필드 없음 | 1차에는 `저장된 히트맵 결과 없음`, 계약 확장 후 표시 |
| 오버레이 영상 | 저장 가능 | PDF 제외, 내부 상세 전용 |
| 신뢰도 정의 | 확인 필요 | 정의가 검증되기 전 PDF에서 제외 |
| 종합 반영 가중치·기여도 | 확인 필요 | 임의 추정 금지, `미제공` 처리 |
| 실제 CoC 연속 이력 | PDF 데이터에 미연결 | 독립 섹션 제거 |

### 완료 조건

- 각 PDF 출력 필드에 데이터 출처가 매핑됨
- 점수 단위와 기준값 표시 규칙이 문서화됨
- 확인되지 않은 `confidenceScore`는 PDF 표시 대상에서 제외됨
- 딥페이크·영상 내용 조작 모듈 분류표가 확정됨

## 5. 단계 1A — PDF 발행 차단 오류 수정

### 목적

분석 결과 내용과 무관하게 제출본으로 사용할 수 없는 출력 오류를 먼저 제거한다.

### 작업

- 4페이지 고정 페이지 수 제거
- 실제 렌더링된 물리 페이지 수를 기준으로 페이지 번호 생성
- 승인 정보가 다음 장으로 밀리는 현재 레이아웃 제거
- 푸터 보호영역 확보
- 표 한 행, QR 블록, 책임 정보 블록의 페이지 분할 금지
- 대각선 미리보기 워터마크 제거
- 미리보기 상단 상태 표시와 낮은 불투명도의 배경 표식 적용
- SHA-256을 말줄임표 없이 64자리 전체 표시
- 미치환 `${...}` 변수가 있으면 발행 실패
- 최종본 QR·검증 ID·보고서 번호 누락 시 발행 실패
- 미리보기에는 QR·실제 검증 URL을 생성하지 않음
- 최종 발행본에는 미리보기 표식을 남기지 않음

### DB 변경

없음.

### 테스트

- 미리보기 PDF 페이지 수 확인
- 최종 발행 PDF 페이지 수 확인
- 긴 파일명·긴 부서명·긴 SHA-256 출력
- 승인 정보가 많은 경우
- QR 생성 성공·실패 경우
- 검증 URL 환경변수 누락 경우
- 모든 페이지를 PNG로 렌더링하여 잘림·겹침 확인

### 완료 조건

- 물리 페이지 수와 `X/Y`가 항상 일치
- 푸터 침범과 잘린 표가 없음
- 미치환 템플릿 문자열이 없음
- 미리보기와 최종본 상태가 혼재하지 않음

## 6. 단계 1B — 5페이지 핵심본 재구성

### 6.1 공통 규칙

- 모든 페이지 헤더: 발행 주체, 보고서 번호, 개정 번호, 문서 상태
- 모든 페이지 푸터: 실제 현재 페이지 / 전체 페이지
- 영문 내부 상태 코드는 한국어 표시값으로 변환
- 결측값을 `-` 하나로 통일하지 않음
- 분석 결과와 발행 상태를 분리
- 숫자 점수는 점수 범위와 기준값을 같은 영역에 표시
- 모델별 점수 의미가 다르면 동일한 0~100 막대로 직접 비교하지 않음

### 6.2 결측값 표준

| 표시값 | 의미 |
|---|---|
| 미제공 | 원천 시스템 또는 생성기에 전달되지 않음 |
| 저장된 결과 없음 | 분석은 수행됐지만 해당 세부 산출물이 저장되지 않음 |
| 미적용 | 분석대상 특성상 해당 모듈을 적용하지 않음 |
| 판별 보류 | 입력 품질이나 판단 조건 때문에 판별하지 못함 |
| 분석 실패 | 기술 오류로 분석을 완료하지 못함 |
| 해당 없음 | 해당 보고서에 개념 자체가 적용되지 않음 |

### 6.3 1페이지 — 분석대상 식별 및 종합 결과

페이지 제목: **AI 기반 영상 분석 결과보고서**

섹션:

1. 문서 식별
2. 사건·분석대상 식별
3. 분석 실행 정보
4. 얼굴 합성·교체 의심 분석
5. 영상 내용 조작 흔적 분석
6. AI 기반 종합 분석 결과
7. 판별 보류·추가 검토
8. 축약 AI 한계 고지

필수 필드:

- 보고서 번호·개정 번호·문서 상태
- 사건번호·사건명
- 증거 ID·파일명
- 분석 요청·실행 ID
- 요청·완료 시각·분석 처리 상태
- 두 분석 영역의 처리 상태
- 두 분석 영역의 범주형 결과
- 두 분석 영역의 출력 점수·기준값
- 두 분석 영역의 종합 반영 여부
- 최종 종합 모델·버전·점수·기준값
- 시스템 내부 위험 분류
- 핵심 근거 최대 4개
- 판별 보류·실패 사유
- 추가 전문가 검토 권고 여부

### 6.4 2페이지 — 분석 방법 및 모델별 결과

표 열:

- 영역
- 모듈
- 모델·버전
- 처리 상태
- 모델 출력 점수
- 판정 기준값
- 기준값 초과 여부
- 비고

규칙:

- 딥페이크 영역과 영상 내용 조작 영역을 시각적으로 그룹화
- 본문에는 종합 결과에 관여한 모델 최대 8개와 최종 종합 모델 1개
- 모델별 기준값이 없으면 추정하지 않고 `미제공`
- 모델별 신뢰도는 정의가 검증된 경우에만 표시
- 원시 영문 출력문은 PDF에 표시하지 않음
- 8개를 넘는 모델은 조건부 부록으로 이동

### 6.5 3페이지 — 시간축 및 시각적 근거

구성:

- 전체 영상 길이를 기준으로 하는 통합 타임라인 1개
- 최대 3개 레인: 종합, 얼굴 합성·교체, 영상 내용 조작 흔적
- 중복을 병합한 주요 의심 구간·지점 최대 4개
- 실제 대표 프레임 최대 3개
- 실제 히트맵 최대 3개
- 얼굴 분석 요약

대표 프레임 캡션:

- 시점
- 프레임 번호
- 관련 분석 영역
- 근거 모델
- 모델 출력 점수·기준값
- 원본 프레임·크롭·확대·히트맵 여부

이미지 규칙:

- 원본 화면비 유지
- 임의 보정·미화 금지
- 얼굴 크롭만 표시할 경우 전체 프레임 축소본도 함께 표시
- 히트맵은 `AI 모델 반응 시각화`라고 명시
- 이미지가 없으면 빈 틀 대신 `저장된 대표 프레임 없음`
- 히트맵 데이터가 없으면 `저장된 히트맵 결과 없음`

### 6.6 4페이지 — 기술 메타데이터 및 분석 입력 파일 식별

필수 필드:

- 증거 ID·파일명·파일 크기
- MIME 유형·컨테이너
- 시스템 등록 시각
- 영상 길이·해상도·FPS
- 영상 코덱·오디오 코덱
- 분석 요청·실행 ID
- 시작·완료 시각·시간대
- 모델 세트 또는 분석 패키지 버전
- PDF 템플릿 버전

## 11. 단계 2C — 무결성 결과 스냅샷 및 PDF·검증 페이지 연결

**상태: 구현 완료 · 회귀 및 시각 검수 완료 (2026-07-14)**

### 목적

이미 구현된 증거 매니페스트 서명, CoC 해시 체인, 증거 블록체인 검증 결과를 보고서 발행 시점에 동결하여 PDF에 정확한 명칭으로 표시한다. 최종 PDF 생성 후에만 알 수 있는 보고서 PDF 해시 블록체인 등록 결과는 PDF 내부에 다시 넣지 않고 공개 검증 페이지에서 제공한다.

### 확정 결정

- PDF 파일 내부에 PAdES 등 인증서 기반 전자서명을 새로 구현하지 않는다.
- PDF의 `PDF 전자서명`, `미적용 (도입 예정)` 항목을 삭제한다.
- 현재 백엔드의 X.509 서명은 `증거 매니페스트 X.509 서명`으로만 표기한다.
- `외부 해시 앵커`라는 모호한 명칭을 사용하지 않는다.
- 증거 원본 해시 앵커는 `증거 해시 블록체인 검증`으로 표기한다.
- 보고서 해시 앵커는 `보고서 PDF 해시 블록체인 등록`으로 표기한다.
- 로컬 `simulated` 앵커는 기관 제출본에서 실제 외부 원장 등록으로 표현하지 않는다.
- CoC 전체 이벤트는 PDF 본문에 나열하지 않고 체인 검증 결과와 기록 건수만 요약한다.

### 현재 구현과 명칭 매핑

| 현재 백엔드 기능 | 정확한 의미 | PDF·화면 표시 |
|---|---|---|
| `EvidenceManifestService` 서명 | 증거 매니페스트 JSON의 X.509 서명 | 증거 매니페스트 서명 |
| `CustodyChainVerifier` | 증거 CoC 로그의 연결 해시 재검증 | CoC 해시 체인 |
| `EVIDENCE_HASH` 앵커 | 분석 입력 증거의 SHA-256 원장 등록 | 증거 해시 블록체인 검증 |
| `REPORT_HASH` 앵커 | 최종 발행 PDF SHA-256 원장 등록 | 보고서 PDF 해시 블록체인 등록 |
| `pdfSignatureApplied=false` | PDF 자체 전자서명 미구현 | PDF에서 비노출 |

### 발행 시점 무결성 스냅샷

보고서 생성 과정에서 알림을 발생시키는 검증 API를 재사용하지 않는다. 부작용 없는 읽기 전용 조회를 추가하여 다음 값을 발행 스냅샷 JSON에 저장한다.

- `verifiedAt`: 무결성 결과를 조회한 시각
- `manifestSignatureStatus`: `VALID / INVALID / NOT_FOUND / FAILED`
- `manifestSignatureAlgorithm`: 실제 기록이 있을 때만 저장
- `cocChainStatus`: `VALID / INVALID / NOT_FOUND`
- `cocLogCount`: 발행 시점 CoC 기록 건수
- `cocBrokenAtLogId`: 체인 실패 시 최초 실패 로그 ID
- `evidenceBlockchainStatus`: `MATCHED / MISMATCHED / PENDING / NOT_ANCHORED / FAILED`
- `evidenceBlockchainNetwork`: 실제 앵커가 있을 때만 저장
- `evidenceBlockchainTransactionHash`: 실제 앵커가 있을 때만 저장
- `evidenceBlockchainAnchoredAt`: 실제 앵커가 있을 때만 저장

기존 `report_publication_snapshots`의 JSON 구조를 확장하며 신규 DB 컬럼은 추가하지 않는다. 스냅샷 스키마 버전은 `1.1`로 올리고, 발행 후에는 현재 CoC·앵커 상태가 바뀌어도 기존 PDF 표시값을 변경하지 않는다.

### PDF 4페이지

페이지 제목을 `분석 입력 파일 및 무결성 검증`으로 변경한다.

1. 분석대상 파일 정보
2. 분석 실행 정보
3. 분석 입력 파일 식별 해시(SHA-256)
4. 발행 시점 무결성 검증 요약

무결성 검증 요약에는 다음 항목만 표시한다.

- 증거 매니페스트 서명: 유효 / 검증 실패 / 기록 없음
- CoC 해시 체인: 유효 / 검증 실패 / 기록 없음
- CoC 기록 건수
- 증거 해시 블록체인: 일치 / 불일치 / 등록 대기 / 미등록 / 실패
- 검증 수행 시각

`수집·인계·보관 이력 범위` 안내 문단은 삭제한다. 실제 상세 이벤트는 기존 내부 무결성·CoC 화면에서 조회한다.

### PDF 5페이지

`서명 및 외부 앵커 상태` 섹션을 삭제한다.

1. 작성·검토 및 발행 책임
2. 발행 등록 정보
3. QR 및 PDF 동일성 검증
4. 검증 범위 및 AI 분석 한계

최종 PDF 해시와 블록체인 트랜잭션 정보는 PDF 내부에 넣지 않는다. PDF를 만든 후 해시와 트랜잭션이 생성되므로 이를 PDF에 다시 삽입하면 파일 해시가 바뀌는 순환 문제가 발생한다.

### 공개 검증 페이지

다음 결과는 최종 PDF 생성 후 공개 검증 페이지에서 표시한다.

- 업로드 PDF와 발행 PDF의 SHA-256 동일성
- 보고서 PDF 해시 블록체인 등록 상태
- 블록체인 네트워크
- 트랜잭션 ID
- 앵커 등록 시각

사건명, 파일명, 분석관 실명, 수치 점수, CoC 상세 이벤트는 공개하지 않는다. 로컬 `simulated` 모드는 `개발 검증용 기록`으로 표시하거나 운영 공개 화면에서 숨긴다.

### API 정리

현재 공개 검증 응답의 `signatureStatus`, `signatureValid`, `signatureAlgorithm`, `signerCertificateSubject`는 PDF 서명이 아니라 증거 매니페스트 서명 결과다. 다음 필드를 추가하고 기존 필드는 한 호환 기간 동안 deprecated 처리한 뒤 제거한다.

- `evidenceManifestSignatureStatus`
- `evidenceManifestSignatureValid`
- `evidenceManifestSignatureAlgorithm`
- `evidenceManifestSignerCertificateSubject`

`pdfSignatureApplied`는 신규 화면에서 사용하지 않고 제거 대상으로 표시한다.

### 구현 순서

1. 부작용 없는 무결성 요약 조회 서비스 추가
2. 발행 스냅샷 스키마 `1.1` 및 무결성 JSON 저장
3. PDF 4페이지 무결성 요약 연결
4. PDF 5페이지 전자서명·외부 앵커 행 제거
5. 공개 검증 API 필드 명칭 보정
6. 프론트 공개 검증 페이지에 보고서 블록체인 상태 연결
7. PDF·API·스냅샷 회귀 테스트 및 5페이지 시각 검수

### 테스트

- 매니페스트 서명 유효·실패·기록 없음
- CoC 체인 유효·손상·기록 없음
- 증거 블록체인 일치·불일치·대기·미등록·실패
- 보고서 블록체인 등록 성공·대기·실패
- `simulated`와 실제 HTTP 앵커 표시 구분
- 발행 후 CoC 로그가 추가되어도 기존 PDF 스냅샷이 변하지 않음
- 공개 검증 페이지에 민감정보가 노출되지 않음
- PDF에 `PDF 전자서명`, `미적용 (도입 예정)`, `외부 해시 앵커`가 남지 않음

### 완료 조건

- PDF의 서명·블록체인·CoC 용어가 실제 구현과 일치
- PDF 4페이지가 발행 시점 무결성 스냅샷만 사용
- PDF 5페이지에 PDF 전자서명 또는 사후 생성되는 트랜잭션 정보가 없음
- 공개 검증 페이지에서 최종 PDF 해시와 보고서 블록체인 기록을 검증 가능
- 발행본 재생성 시 동일한 무결성 스냅샷 사용

### 구현 결과 (2026-07-14)

- `ReportIntegritySnapshotService`를 추가해 보고서 생성 중 보안 알림을 만들지 않는 읽기 전용 검증을 수행한다.
- 분석 보고서 발행 스냅샷을 `1.1`, PDF 템플릿을 `analysis-5p-v3`으로 올렸다.
- PDF 4페이지에 매니페스트 서명, CoC 해시 체인·기록 건수, 증거 해시 블록체인, 검증 시각을 표시한다.
- PDF 5페이지와 비교검증 보고서에서 PDF 전자서명·외부 해시 앵커 예정 문구를 제거했다.
- 공개 검증 API에 `evidenceManifestSignature*` 필드를 추가했고, 기존 모호한 `signature*`와 `pdfSignatureApplied`는 호환용으로만 유지한다.
- 공개 검증 화면은 최종 PDF SHA-256 등록값과 보고서 PDF 해시 블록체인 등록을 구분하며, `local-simulated`는 개발 검증용 기록으로 표시한다.
- 신규 DB 컬럼이나 마이그레이션은 추가하지 않았다.

## 12. 단계 3 — 기관 양식·선택적 서명·상세 이력 확장

이 단계는 기술팀이 단독으로 결정하지 않는다.

### CoC 상세 확장

- 현재 CoC 로그보다 상세한 수집·수령·복제·반출·반환·폐기 업무 절차가 필요한지 기관과 확정
- 위치·확인 방법·인수자 서명 등 현재 없는 필드가 필요한 경우 별도 데이터 계약 추가
- 상세 이력 부록은 의뢰기관이 요구하고 공개 범위가 확정된 경우에만 생성

### 외부 앵커

- 실제 네트워크·서비스·거래 ID·시각·대상 해시 저장
- 기술적 존재 사실만 표시
- 법적 효력 자동 보장 표현 금지

### 선택적 PDF 전자서명

- 수신기관이 서명된 PDF를 명시적으로 요구하는 경우에만 PAdES 등 적용 수단과 검증 방식 확정
- 수신기관의 서명·날인 요구 확인
- 서명 스캔본은 원 PDF와 다른 파일로 별도 해시·버전 관리

### 기관 양식

- 법원·검찰·경찰·기타 의뢰기관별 필수 항목 확인
- 문서 제목, 사건번호, 작성자 자격, 서명·날인, 제출 매체 규칙 확인
- 기관별 템플릿을 기본 보고서와 분리

## 15. 발행 차단 검사

다음 중 하나라도 발생하면 최종 발행을 허용하지 않는다.

- 물리 페이지 수와 `X/Y` 불일치
- URL 또는 템플릿 변수가 치환되지 않음
- 보고서 번호·개정 번호 누락
- 분석 입력 파일 SHA-256 누락 또는 잘림
- 분석 상태와 출력 결과가 모순됨
- 미리보기 표식이 최종본에 남음
- 최종본 QR 또는 검증 ID 누락
- 승인·QR·AI 한계 고지 블록 잘림
- 빈 페이지 발생
- 실제 데이터가 없는 CoC·앵커 값 생성
